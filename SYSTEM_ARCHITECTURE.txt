4. AUTHENTICATION & SESSIONS
================================================================================

4.1 Login flow
--------------
API: POST /api/v1/accounts/auth/login/   (AllowAny)

1. Client sends { identifier, password? }
   - Identifier = email OR index number (normalized)
2. If eligible index has no User → auto-create student account
3. If staff (is_staff / is_superuser / role in admin|super_admin|auditor):
   - Password required first (response may say requires_password)
4. OTP always issued via OTPService
5. Response returns otp_session_id (OTP may print to console in dig)

Frontend: LoginView → authStore.login() → navigate /otp?sessionId=


4.2 OTP verify → JWT
--------------------
API: POST /api/v1/accounts/auth/otp/verify/

Returns access + refresh tokens via SessionService (SimpleJWT).
Frontend stores in localStorage:
  access_token, refresh_token, user_uuid, session_uuid, user_role,
  requires_onboarding, is_new_user

JWT config (backend settings):
  Access:  ~15 minutes
  Refresh: ~7 days
  Claim uses user uuid

Axios clients attach Bearer; on 401 try refresh at
  POST /api/v1/accounts/auth/token/refresh/
  Fail → clear tokens → login


4.3 Logout
----------
POST /api/v1/accounts/auth/logout/
Revokes custom Session by JTI; attempts token.blacklist()
NOTE: token_blacklist app may not be installed — blacklist call may be partial.


4.4 Student onboarding
----------------------
Gate: onboarding_completed == false for student/candidate
Forced route: /onboarding
API:
  GET  /api/v1/accounts/onboarding/options/  → faculties, depts, levels
  POST /api/v1/accounts/onboarding/          → profile + academic FKs

Blocks voting until completed + scope eligibility checks pass.


4.5 Current user
----------------
GET /api/v1/accounts/auth/me/

8. VOTING DOMAIN (STUDENT CORE FLOW)
================================================================================

8.1 End-to-end student journey
------------------------------
1. Login (index) → OTP → JWT
2. Onboarding if required
3. /student — list eligible open elections (and already-voted)
4. Enter vote → /vote/:uuid
5. Request & validate SVT (Secure Voting Token) format v-xxx-####
6. /vote/:uuid/presence — capture presence selfie
7. /vote/:uuid/ballot — position wizard + BallotBoxCeremony animation
8. Submit votes → confirmation code
9. /vote/:uuid/confirmation — receipt

Step guards (client): already voted → confirmation; SVT invalid → entry;
presence incomplete → presence page.


8.2 Backend voting APIs (/api/v1/voting/)
----------------------------------------
GET  eligible/
POST elections/<uuid>/svt/request/
POST elections/<uuid>/svt/validate/
GET  elections/<uuid>/svt/session/
GET/POST elections/<uuid>/presence/
GET  elections/<uuid>/ballot/          (requires validated SVT + presence)
POST elections/<uuid>/submit/
POST elections/<uuid>/voters/<user_uuid>/clear/   AdminOrSuperAdmin (test)

All vote path views: IsAuthenticated + business rules in view.


8.3 Business rules on submit
----------------------------
Prerequisite chain:
  - Authenticated student
  - Onboarding completed
  - On VoterEligibility list
  - Matches election academic scope
  - Election status open; web voting allowed
  - SVT validated (then marked used on submit)
  - Presence capture exists for (user, election, svt)
  - Candidates must be approved; respect max_votes_allowed per position

On success:
  - Create Vote rows (user + election + position + candidate + channel + SVT)
  - Per-vote vote_hash (SHA-256)
  - Unique constraint (user, position, candidate)
  - Mark SVT used
  - Write vote_cast AuditLog WITHOUT ballot choices (privacy sanitize)
  - Return confirmation code

NOTE: Votes retain user FK — DB is NOT cryptographically anonymous.
      Ballot secrecy is for audit metadata, not full anonymization.


8.4 SVT (security app)
----------------------
Model SVTToken:
  hashed token, status issued|validated|used|expired|revoked,
  attempt counters

Settings-driven (system/settings_utils.py):
  svt_max_requests_per_hour
  svt_resend_cooldown_seconds
  svt_max_validation_attempts
  (svt_expiry_minutes deprecated — lives until election end / use)

Client caches SVT state: sessionStorage key svt:{electionUuid}


8.5 Presence
------------
Model PreVotePresenceCapture — selfie before ballot
Linked on AuditLog via presence_capture_id (not joined to vote choices)


8.6 Vote model
--------------
Vote: user, election, position, candidate, channel, optional SVT,

Unique (user, position, candidate)

20. SECURITY MODEL (SUMMARY)
================================================================================

Layer                    Mechanism
-----------------------  -----------------------------------------------------
Transport                HTTPS optional via vite LAN/HTTPS scripts
Auth                     JWT Bearer (SimpleJWT)
Staff gate               Password then OTP
Student gate             Index + OTP (auto-provision if eligible)
OTP at rest              SHA-256 hashed
SVT at rest              SHA-256 hashed; rate limits; attempt revoke
Vote integrity           per-vote SHA-256 vote_hash
Result integrity         result_hash of standings snapshot
Audit ballot secrecy     metadata sanitization (no choices in audit)
Strongroom               password step-up → vault token TTL
Public verify            seal hash existence check
Device context           optional DeviceLog fingerprint from client
DRF throttle             anon 100/hour, user 1000/hour (defaults)
CORS                     allow-all typical in dig
Settings secrets         is_encrypted flag (masking; not full cipher vault)

Known gaps / caveats:
  - Votes store user FK (not anonymous ballots)
  - OTP/SVT plaintext may print in dig console
  - max_login_attempts / OTP attempt counters under-enforced
  - JWT blacklist without blacklist app installed
  - IsAdmin excludes super_admin from EC APIs
  - Fraud detection not a continuous automated pipeline
  - USSD public ingress incomplete relative to models
  - Channels installed but unused

23. END-TO-END SCENARIOS (HOW THE SYSTEM WORKS)
================================================================================

A) Fresh student votes
----------------------
1. SuperAdmin ensures academic structure + settings + web_voting flag
2. Admin creates election (scope), positions, candidates (approve),
   imports/adds eligibility
3. Admin opens election
4. Student logs in with index → OTP → completes onboarding if needed
5. Student sees election on /student → SVT → presence → ballot → confirm
6. Monitor shows live turnout (admin/auditor)
7. Admin closes election → generates → certifies → publishes results
8. Strongroom seals available for custody/reveal
9. Student can open published result detail

B) Auditor oversight
--------------------
Same as admin for viewing elections, results, strongroom, monitor, audit
Cannot create/manage/open/close, certify, fraud, or USSD

C) Super admin platform change
------------------------------
Users, academic tree, theme, feature flags, security OTP/SVT knobs,
maintenance mode, institution branding, operations health
Does NOT drive election day ops unless also given admin role

D) Strongroom custody
---------------------
Admin/auditor unlock vault with password → timed token
Access sealed elections → reveal seal hashes (logged evidence)
Optionally lock; anyone can verify a seal hash publicly via API

25. MENTAL MODEL FOR CHANGES
================================================================================

Ask first: which ROLE is affected?
  student/candidate → voting + student routes + eligibility/scope
  admin (EC)        → elections/candidates/voters/results/fraud/ussd/monitor
  auditor           → same views, readonly; permission classes viewer/*
  super_admin       → users/academic/settings/ops/theme — NOT IsAdmin APIs

Then ask: which LAYER?
  Privilege gate → accounts/permissions.py AND frontend router + Sidebar
  Business rule  → domain services / api/views.py
  UI flow        → views + components under frontend/src
  Config knob    → SystemSetting / FeatureFlag + settings_utils

Remember split:
  Platform governance ≠ Election Committee
  Frontend role menus and backend IsAdmin / IsSuperAdmin must stay aligned
  When adding a feature, update: model → serializer → view permissions →
  urls → frontend api module → route meta → sidebar (if nav) → views
