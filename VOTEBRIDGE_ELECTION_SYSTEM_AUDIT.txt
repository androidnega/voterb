1.1 Election model
------------------
File: backend/elections/models.py (class Election, ~lines 44–123)

Fields:
  id                      BigAutoField (PK)
  uuid                    UUIDField (unique, non-editable)
  title                   CharField(255)                         REQUIRED
  description             TextField (blank=True)                 OPTIONAL
  election_type           CharField choices TYPE_CHOICES         default='general'
  status                  CharField choices STATUS_CHOICES       default='draft'
  start_date              DateTimeField (null/blank)             OPTIONAL at model;
                                                                 REQUIRED by create form UX
  end_date                DateTimeField (null/blank)             OPTIONAL at model;
                                                                 REQUIRED by create form UX
  allow_web_voting        BooleanField                           default=True
  allow_ussd_voting       BooleanField                           default=False
  allow_sms_notifications BooleanField                           default=False
  scope_type              CharField SCOPE_CHOICES                default='school'
  scope_faculty           FK Faculty (SET_NULL, null/blank)
  scope_department        FK Department (SET_NULL, null/blank)
  scope_level             FK Level (SET_NULL, null/blank)
  created_by              FK User (PROTECT)                      set on create
  demo_seed               BooleanField                           default=False
  created_at / updated_at auto timestamps

STATUS_CHOICES:
  draft | scheduled | open | paused | closed | archived

TYPE_CHOICES (election_type):
  general | student_union | faculty | departmental | special

SCOPE_CHOICES (scope_type):
  school | faculty | department | level

Helper methods on Election:
  user_matches_scope(user)
    - school → True for everyone
    - faculty → user.faculty_id == scope_faculty_id
    - department → user.department_id == scope_department_id
    - level → user.level_id == scope_level_id

  get_scope_eligible_students()
    - Returns active onboarded students filtered by scope FKs

1.3 Where scope is used
-----------------------
  backend/elections/models.py
    - scope_type + scope_* FKs + user_matches_scope / get_scope_eligible_students

  backend/elections/serializers.py (ElectionCreateUpdateSerializer)
    - Validates required FK for faculty/department/level scopes
    - Resolves *_uuid write fields to FKs on create/update

  backend/elections/services/scope_eligibility.py
    - student_can_access_election(user, election):
        1) user active
        2) student must have onboarding_completed
        3) VoterEligibility row exists (is_eligible=True)
        4) election.user_matches_scope(user)

  backend/voting/api/views.py
    - EligibleElectionsView + SVT request path call student_can_access_election

  frontend CreateElectionForm.vue
    - Collects scope_type + conditional faculty/department/level

IMPORTANT COMBINED RULE:
  Scope alone does NOT put someone on the ballot. They must ALSO be on the
  election's VoterEligibility list. Scope is an ADDITIONAL filter.

1.5 Required vs optional (create)
---------------------------------
Frontend-enforced:
  REQUIRED: title, start_date, end_date, end > start
  REQUIRED when scope_type=faculty: scope_faculty_uuid
  REQUIRED when scope_type=department: scope_department_uuid
  REQUIRED when scope_type=level: scope_level_uuid
  OPTIONAL: description, election_type (defaults), channels, allow_ussd

Backend serializer (ElectionCreateUpdateSerializer):
  REQUIRED: title (model)
  OPTIONAL: description, dates, channels, election_type, status
  CONDITIONAL: scope FKs based on scope_type
  created_by: set in perform_create, not client-supplied

2.2 Is there a "Voter Register" or "Category"?
---------------------------------------------
NO dedicated VoterRegister / VoterCategory / RegisterCategory model exists.

Closest concepts today:
  - VoterEligibility = flat per-election voter list
  - User.faculty / User.department / User.level = academic profile
  - Election.scope_* = election-wide academic filter

There is NO:
  - Named voter register entity
  - Category / ward / college / hostel grouping for eligibility
  - Position-linked voter category


2.3 How voters are imported
---------------------------
Backend: backend/elections/api/eligibility_views.py → EligibilityBulkImportView
Frontend: frontend/src/components/elections/VoterManager.vue (Import dialog)
API: POST /api/v1/elections/<election_uuid>/eligibility/import/
Permission: IsAdmin

CSV columns:
  REQUIRED:
    index_number  (or "identifier")
    faculty_code  (or "faculty") — unless user already has faculty+department
    department_code (or "department")
  OPTIONAL:
    first_name, last_name
    level / level_name

Example (from VoterManager UI):
  index_number,faculty_code,department_code,level
  SC/2021/PL/001,ENG,PL,Level 400

Behavior:
  - Resolves/creates student by index via resolve_or_create_voter()
  - Applies academic placement via apply_student_academic_placement()
  - Creates VoterEligibility(is_eligible=True, verified_by=request.user)
  - Skips duplicates; collects per-row errors
  - Rejects email-style identifiers


2.4 Manual add
--------------
POST /api/v1/elections/<election_uuid>/eligibility/
Serializer: VoterEligibilitySerializer
Requires: user_identifier (index) OR user_uuid
Requires: faculty_uuid + department_uuid
Optional: level_uuid
Sets verified_by on create


2.5 Grouping beyond faculty/department/level?
---------------------------------------------
NO. Grouping is only:
  User academic FKs + election scope + flat eligibility list.
eligibility_reason exists on the model but is not meaningfully used in UI flows.

5.3 Checks before ballot is shown (backend BallotView)
------------------------------------------------------
File: backend/voting/api/views.py class BallotView

  1) election.status == 'open'
  2) user has not already voted (user_has_voted)
  3) active validated SVT exists
  4) PreVotePresenceCapture image exists for that SVT/election
  5) Then returns positions where is_active=True AND is_votable=True
     with candidates status='approved' ordered by ballot_number

Earlier gate (EligibleElectionsView / SVTRequest):
  student_can_access_election → eligibility + onboarding + scope

5.5 SubmitVoteView checks
-------------------------
  - Validated SVT (optional code match)
  - Election open
  - Not already voted
  - Presence captured
  - Position is_votable
  - len(candidate_uuids) <= max_votes_allowed
  - Candidates approved for that position
  - Creates Vote rows, marks SVT used
  - record_vote_cast_audit(...) — device/IP/location/fingerprint for VOTER audit
  - Returns confirmation_code

6.1 ElectionResult model
------------------------
File: backend/results/models.py

STATUS_CHOICES:
  generated | pending_certification | certified | published | archived

Fields:
  uuid
  election           OneToOneField Election
  status             default='generated'
  standings          JSONField  { positions: [{ title, candidates: [...] }] }

  turnout_percentage Decimal
  certified_by       FK User null
  certified_at       DateTime null
  published_at       DateTime null
  created_at / updated_at


6.2 Certification process
-------------------------
Flow:
  1) Election closed
  2) POST /results/elections/<uuid>/generate/  (IsAdmin)
       → creates ElectionResult status=generated
  3) POST /results/elections/<uuid>/certify/   (IsAdmin)
       → status=certified, sets certified_by + certified_at
  4) POST /results/elections/<uuid>/publish/   (IsAdmin)
       → status=published, sets published_at

Frontend:
  ElectionDetailView: Generate results button when closed
  ResultsListView / ResultDetailView: Certify + Publish buttons

pending_certification status exists in choices but generate() writes 'generated'
directly; certify accepts generated OR pending_certification.


6.3 EC verification step (photo / location / IP / fingerprint / signature)?
---------------------------------------------------------------------------
For RESULT CERTIFICATION: NO.
  CertifyResultsView only sets status + certified_by + certified_at.
  No photo, GPS, IP, device fingerprint, or digital signature captured on certify.

For VOTER vote-cast: YES (separate concern).
  security/services/vote_audit.py record_vote_cast_audit stores:
    device fingerprint, device_type, OS, IP, geo location, presence photo link
  This is voter audit trail, NOT EC certification of results.

   IMPLEMENTED:

   IMPLEMENTED:
     - Results preview for auditors (generated results)
     - Clear student vote (admin)
     - demo_seed flag for seeded elections
   MISSING:
     - EC dummy vote / dry-run ballot
     - Preview mode that does not write real Vote rows

E) EC certification with photo / location / IP / fingerprint / signature
   IMPLEMENTED:

     - EC certify ceremony capturing photo
     - EC certify capturing location / IP / fingerprint
     - Digital signature / multi-officer co-certification
     - pending_certification workflow as a distinct mandatory step
     - Binding certification evidence into ElectionResult integrity_report

10. CURRENT ELECTION CREATION → VOTE → RESULTS CODE PATH (MAP)
================================================================================

CREATE
  ElectionListView "New Election"
    → CreateElectionForm.vue
    → POST /api/v1/elections/
    → ElectionListCreateView.perform_create(created_by=user)
    → status=draft

CONFIGURE (tabs on ElectionDetailView)
  Positions → POST .../positions/
  Candidates → POST .../candidates/ (+ approve)
  Voters → POST .../eligibility/ or .../eligibility/import/

LIFECYCLE
  draft → Schedule (status update) → scheduled
  scheduled → POST .../open/ → open
  open/paused → POST .../close/ → closed

VOTE (student)
  eligible list → SVT request/validate → presence photo → ballot → submit
  gates: eligibility + scope + SVT + presence + open status

RESULTS
  closed → generate → certify → publish
  live monitor available while open/paused/closed/scheduled via monitor APIs

11. GAPS / TECH NOTES RELEVANT TO RESTRUCTURE
================================================================================

1. election_type is cosmetic; do not assume it drives eligibility.
2. Scope + eligibility are AND-gated for students.
3. Positions cannot target subsets of the voter list.
4. No Position update/delete API — only list/create.
5. No /elections/create route — modal-only create UX.
6. Positions/candidates/voters are tabs, not nested routes.
7. Certification is a one-click admin action, not an evidence ceremony.
8. is_votable is in the model/API but not in the Add Position UI.
9. Token refresh endpoint was previously missing (auth 401s) — separate from
   election domain but affects EC ability to create positions while working.
